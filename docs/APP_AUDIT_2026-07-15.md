# Summer Rescue app audit and implementation record — deep pass, 15 July 2026

## Executive verdict

The app is a sophisticated campaign-specific operating system rather than a shallow prototype. It contains 20 purposeful destinations, specialist module workspaces, a separate Life/Health/Admin area, a capacity-aware schedule, evidence-based completion, 200 verified local resources, and strong pure-logic tests. That functional breadth is preserved.

The initial problem was not excessive capability. It was that the persistence layers, privacy boundary, large collections, overlays, and advanced Focus subsystems were not reliably integrated with the capability already present. The implementation therefore stabilizes and connects the rich product rather than reducing it.

This pass inspected every top-level route, module Overview/Planning/Materials, desktop and mobile layouts, settings, source boundaries, live JSON/SQLite/event state, API behavior, Docker/native startup, repository privacy, resource rendering, tests, assets, lint and build. It then implemented the campaign-safe architecture described in `PROPOSED_ARCHITECTURE.md` while deliberately deferring the post-campaign SQLite-authority migration.

## Product and UX audit

### P0 correctness problems

1. **The pre-campaign Today copy is factually wrong.** On Wednesday 15 July, it says “Preview Monday’s protected launch schedule” even though the launch preview is Thursday 16 July. The string is hard-coded.
2. **MAT700 cannot be dropped through the app.** Migration always writes `mat700Active: true`, and Settings renders a checked, read-only, disabled checkbox. This contradicts the campaign rule that MAT700 should be removed if the result is safely passed.
3. **The intro can render before file-backed state hydration.** On a cold load it can briefly show default totals before the saved JSON replaces them, so the first screen is not guaranteed to be truthful.
4. **Opening a new app session rewrites the JSON and SQLite files even without a semantic edit.** `savedAt` starts empty; once the file is read, the save effect schedules a whole-state write. Inspection alone advanced file modification times.

### Information architecture

The sidebar intentionally preserves 20 top-level destinations:

- 6 Study routes;
- 5 Planning routes;
- 4 Board routes;
- 5 Focus/administrative routes, including the non-academic Life/Health/Admin area.

Several are alternate projections of the same 116 cards, but they now have explicit purposes: Schedule owns capacity boundaries and durable Done/Skipped block logs, Planner owns forward workload, Progress owns trajectory and drill-down, Analytics owns pattern diagnosis, Columns owns flow, Table owns dense editing, Week owns temporal execution, Evidence owns outputs, Life/Health/Admin owns non-academic routines, and Rescue owns exceptions. The design requirement is legibility and connected state, not fewer capabilities.

The module pattern is strong, but each module then adds Overview, Planning and Materials. MAT700 Planning rendered 30 articles, 96 buttons and 72 inputs across 6,553 px; Materials rendered 59 articles for 51 resources across 4,577 px. “Start here” and “Recently opened” can duplicate resources already present in groups.

### Today

Today exposes a current boundary, ranked outputs, finish/evidence requirements, complete day schedule, yesterday context, reviewed re-plan controls, focus forest, achievements, adjustable daily goal, streak, priority allocation, wrap-up, and connected-workspace shortcuts. This density is deliberate. Expandable analytical sections and bounded collection rendering preserve access without deleting functionality.

### Board and table

The original implementation rendered the entire card collection. The implementation now uses bounded pagination while preserving all records and editing capabilities.

The full capability set remains available on mobile. The navigation moves behind a menu, the document body does not overflow horizontally, and dense data remains inside bounded/paged control surfaces.

### Settings and overlays

Settings, Add Card, and Import Preview now use a shared accessible dialog primitive with initial focus, focus trap, Escape close, focus return, and body scroll lock. The responsive Add Card action has an explicit accessible name.

### Terminology and product contradictions

- Settings and Data Health now identify JSON as authoritative, SQLite as a projection, and the event log as partial audit history.
- Planner, Schedule, Week, Progress, and Analytics retain separate operational definitions in the architecture document.
- Levels, points, streak, achievements, and planting remain first-class focus features with explicit Strict/Gentle user choice.
- Module pages expose one global H1 and subordinate workspace headings.

## Current-product comparison

These comparisons identify useful interaction patterns, not a mandate to imitate minimalist products. Summer Rescue deliberately retains a broader, more technical control surface.

### Sunsama

Sunsama separates Focus Mode from Daily Planning. Summer Rescue keeps both the rich Today cockpit and its dedicated full-screen Focus Room so the user can choose between overview and immersion without losing either capability.

### Motion

Motion schedules from duration, start date, deadline, priority, effort, working schedules and calendar availability, then adjusts as inputs change. Summer Rescue's re-plan is deterministic, revisioned, backed up, undoable, phase-ordered, and restricted to protected academic capacity. Work that cannot fit before 16 August remains unscheduled Backlog rather than receiving fabricated post-window dates.

### Structured

Structured treats Replan as a specific recovery action. Summer Rescue’s one-card-at-a-time rescue triage remains a strong permanent capability, supplemented by reviewed bulk re-planning for a different scale of failure.

### Things

Things distinguishes availability from deadlines. Summer Rescue keeps its richer evidence, resources, schedule boundaries, and permanent specialist views while retaining that date distinction where it improves planning accuracy.

## Synchronization and persistence

### What propagates correctly in one tab

Cards, custom cards, module notes, notifications, resource review state, uploaded-resource metadata, recent resources, snapshots, day logs and campaign settings share the central tracker state. Card completion/status/hours therefore update Today, module pages, board/table/week, progress and notifications together in the same app instance.

The central state mirrors to browser storage and, after an 800 ms debounce, a JSON file. The server then projects that envelope into SQLite. Selected card actions additionally append typed events.

### Unsynchronized islands

- re-plan configuration;
- sidebar and intro preferences;
- per-Today disclosure preferences;
- rolling browser safety copies.

These remaining islands are intentionally ephemeral UI preferences or bounded safety copies. Focus points, streaks, trees, levels, achievements, daily goals, and Strict/Gentle mode are now mirrored into the revisioned JSON state, portable exports/imports, cross-tab refresh, and the SQLite settings projection.

### Concurrency

The JSON envelope now carries a monotonic revision, writer ID, and ETag. `PUT /api/state` requires `If-Match`; stale tabs receive `409` instead of overwriting. Successful writes notify same-origin tabs through `BroadcastChannel`, and semantic no-ops do not rewrite disk.

### Dangerous bootstrap edge case

If the local API returns 204 while browser tracker state exists, the hook now preserves and promotes the browser copy rather than resetting it.

### Exam-date confidence model — implemented 16 July

The exact personal sitting dates for CMT307, MAT508 and MAT700 remain unknown and are not invented.
`src/utils/examDates.js` now owns exam identity and confidence for every consumer:

- `confirmed` — a validated personal date is recorded for that module;
- `unconfirmed` — no personal date is known; the module renders an explicit unconfirmed badge;
- `window` — the countdown is running against the 17–28 August window placeholder and says so.

The window start is never promoted into a per-module date; a test asserts this directly. Entry is
validated in Settings: malformed and pre-campaign dates are always rejected, and an out-of-window
date is only stored after an explicit "record anyway" confirmation, after which it stays flagged as
an exception. Clearing a date removes the key entirely and restores the unconfirmed state, so an
absent key and an empty string can never mean different things. A stored value that fails validation
is reported unconfirmed rather than rendered as fact.

Connected consumers: countdown, Today hero badge, Admin & Dates exam watch, Schedule exam strip and
day badges, and the countdown notifications, which now say "days to the exam window" until a real
date exists instead of "days to exams".

`ScheduleView` previously carried a private module-label map that had already drifted (`Data Mining`
where every other surface said `MAT700`); it now reads the shared model. The schedule exam chip
therefore reads `MAT700`. `utils/notifications.js` had the only extensionless relative import in
`src/utils`, which is why it had no test coverage; adding the extension made it testable under
`node --test`.

### Event coverage — measured, not asserted

`src/state/eventCoverage.js` records the coverage matrix for all 26 mutation families, and
`tests/eventCoverage.test.mjs` verifies it against the real emission sites in `useTrackerState.js`:
every declared event must actually be emitted, every emitted event must be declared, and every named
mutation must be exported. A new mutation that forgets its event fails the suite rather than quietly
eroding recovery.

The matrix tracks two independent questions, because conflating them is how an audit log starts
lying:

- `status` — is the change written down at all?
- `replayed` — can `rebuildProgressFromEvents` actually reconstruct it?

An event can be perfect audit history and still rebuild nothing. `card.deleted` is exactly that: the
tombstone is emitted, but replay has no case for it, so a replayed card comes back.

Measured before this pass: 10 families emitted, 1 partial, 15 missing.
**Measured now: 18 of 26 families emitted, 9 of 26 replayable.** Full replay remains unsafe and Data
Health reports both numbers separately.

Closed in this pass:

- `updateCardDetails` now emits `card.details_edited` as a field-level diff, which brings **phase,
  module/area, priority, slot and estimated hours** into the log. The form submits every field on
  every save, so the payload is diffed against the live card and only real movement is recorded;
  `to` holds absolute values so replay stays idempotent. The diff is `src/utils/cardDiff.js`, kept
  pure and tested rather than buried in the hook closure.
- `rescheduleCards` and `applyReplanSchedule` now emit `card.rescheduled` per card, so bulk
  re-planning is auditable. Both mirror the skip filter of their own updater, so the log never claims
  to have moved a card the updater left alone.
- `resetCardState` now emits a `card.progress_reset` tombstone, and replay honours it by clearing
  accumulated progress while keeping card identity.

Still not replayable, and therefore still not recovery:

- card details, dates, phase, priority, slot and estimated hours are written but have no column in
  the replay model — they are audit history only. Closing this is a projection/schema change, and
  belongs with the SQLite work rather than being half-done here;
- `card.deleted` needs a product decision first: does deleting a catalogue card remove the card, or
  only its progress overlay?
- evidence attachments ride inside `evidence.added`, but replay keeps only id and text;
- day logs, campaign settings, module exam dates, focus rewards, module notes, notifications,
  uploaded resources and whole-state import/reset remain JSON-only.

Verified against the live API: a `card.details_edited` event posts, checksums, appends, and is
deduplicated on a repeated id (41 lines, 1 occurrence). The synthetic verification event was then
removed and the log confirmed byte-identical to its pre-test backup.

### SQLite and event log are not independent recovery layers

- JSON is explicitly the primary store.
- SQLite is a clean projection after accepted revisioned saves; Focus rewards are included in its settings projection.
- The event log covers selected mutations and is explicitly labelled partial audit history.
- Current replay still cannot reconstruct every domain field, so it is not presented as independent recovery.
- Event appends are serialized, checksummed, idempotent, retried sequentially, and drained with `sendBeacon` on page hide.
- Malformed fragments are quarantined and no longer break valid event history.

The system has redundancy, but no defined consistency or recovery guarantee.

## Infrastructure and security

### Repository privacy — resolved P0

Both pre-reset backup files were committed to Git:

- `backup_tracker_state_before_16july_reset_20260715.json`
- `backup_app_sqlite_before_16july_reset_20260715.sqlite`

They are now covered by ignore rules and removed from Git tracking while the physical local backups remain intact. If an earlier commit left the machine, repository-history cleanup remains a separate decision.

### Docker privacy — resolved P0

The earlier `.dockerignore` excluded only `node_modules`, `dist`, `.git` and logs. It now excludes:

- live `local-data` tracker state and SQLite;
- personal backups and tracker exports;
- 131.6 MB of local study assets;
- local QA captures and other state patterns.

Runtime bind mounts still provide the required local state, but private files are no longer sent in the image build context.

### Network boundary

- Native production and Docker publish on `127.0.0.1`.
- The API validates local Host/Origin/Sec-Fetch context.
- Database rebuild is POST-only and requires a fresh one-use token.
- JSON, resource, and shell responses apply consistent `nosniff`, referrer, permissions, and CSP headers.

### Resource rendering

Local/uploaded HTML now runs without script or same-origin privilege and receives a restrictive sandbox CSP. The external Google Fonts dependency was removed.

### Runtime reliability

- Native startup uses a hidden scheduled task and a prebuilt `dist`; it does not rebuild automatically.
- The scheduled task has no explicit restart-on-failure policy.
- Docker and native production are different stacks with different host exposure, headers, hot reload and build freshness.
- The app has no explicit database integrity check or verified rotating backup process.

## Maintainability

- `src/`: roughly 32,000 lines and 1.0 MB.
- `App.jsx`: roughly 77 kB; routing, shell and orchestration are centralized.
- `App.css`: roughly 152 kB and over 7,600 lines.
- `useTrackerState.js`: roughly 44 kB and combines state migration, domain mutations, persistence and event emission.
- Static card/resource catalogues are large JavaScript bundles.
- No route-level code splitting exists.

Tests are strong for pure scheduling, migration, persistence and resource integrity. Missing coverage includes component interactions, accessible dialogs, visual regression, cross-tab conflict, corrupt live event logs, bootstrap 204 behavior, no-op session writes, Docker privacy and failure-injection across JSON/SQLite/event persistence.

## Prioritized risk register after implementation

### Resolved campaign-safety risks

1. Private backups are ignored and untracked; Docker context excludes state, backups, and study assets.
2. Malformed event history is repaired/quarantined and visible in Data Health.
3. MAT700 can be paused and the choice survives migration.
4. Revision conflicts prevent silent cross-tab overwrite; semantic no-ops do not rewrite storage.
5. Dynamic Today date language replaces the hard-coded weekday.
6. Docker/API requests are local-only; rebuild is protected; resources are sandboxed.
7. Dense collections are paged; shared dialogs meet the implemented accessibility contract.
8. Planting, points, streaks, levels, achievements, daily goals, toasts, and Strict/Gentle Focus guard are restored and synchronized.

### P1 — next safe upgrades

1. Add component tests for Today, Focus Room, module workspaces, Settings, and dialogs. No browser
   automation is installed in this repository, so no route-level interaction test exists yet; the
   exhaustive click/responsive matrix remains unexecuted.
2. Add automated accessibility and desktop/mobile visual regression coverage.
3. Add a visible verified-backup timestamp and domain parity report to Data Health. Data Health now
   reports last valid event and per-family coverage; the backup timestamp remains outstanding.
4. Add route-level lazy loading without hiding or removing any destination. **Done, partially.** All
   20 destinations remain; Today stays eager and 8 heavy surfaces (module workspaces, resource
   reader, TrackerViews, Progress/Charts, Study Hub, Job Hunt, Rescue) now load on first navigation
   behind an accessible `role="status"` fallback. Initial chunk 708.40 → 621.67 kB (−12%).

   It is still over the 500 kB threshold, and the remaining weight is data, not components:
   `baseCards.js` is 134 kB and `summerRescuePlan.js` 45.9 kB, both needed by Today. Splitting the
   card catalogue would mean Today waiting on a fetch to know what to show. Getting under 500 kB
   needs the catalogue restructured or moved behind the API — a real change, not a chunking tweak.
   `ScheduleView`/`DayReview` are marked lazy but stay in the initial chunk because Today imports
   their daily-agenda pieces; that is correct and the lazy wrapper is harmless.
5. Extend typed event coverage where it improves auditability; do not claim full replay until parity
   exists. Emission is now 18/26 after `updateCardDetails`, bulk reschedule, re-plan and card reset
   were wired up. The remaining work is on the **replay** side, not the emission side: 17 families
   are still unrebuildable, and card details/dates need columns in the replay model and SQLite
   projection before their events mean anything for recovery. That is a projection change and
   belongs with the SQLite phase. `card.deleted` needs a product decision before it can replay at
   all. No end-to-end replay-parity test should be written that implies recovery is achievable
   before those gaps close.

2. Split frontend feature ownership and CSS behind behavior-preserving tests.
3. Implement verified rotating backup/restore and a tested rollback observation window.
